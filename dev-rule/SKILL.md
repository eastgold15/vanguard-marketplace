# 开发规则

## 一、抽象原则

### 什么时候做抽象？
满足**任意一条**就值得抽：

- **存在多个可互换的实现**（最核心）\
  比如本地推理 vs 远程 HTTP 推理，两套逻辑对外行为一致、内部实现不同，需要随时切换/兼容。

- **上层不想感知底层细节**\
  UI/业务代码不想写一堆 `if 本地 else 远程`，不想耦合网络、模型加载、推理细节。

- **未来会新增实现**\
  以后可能加边缘节点、离线缓存、第三方 API，抽象后不用改上层代码。

- **需要统一规则、统一入参出参**\
  两套逻辑参数、返回格式不一致，抽象层做统一收口、格式适配。

### 什么时候别抽象？
- 只有唯一实现，未来也不会变
- 逻辑极简单，抽抽象反而增加代码复杂度

**核心原则：简单就直写，多变就抽象。优先函数式。**

## 二、代码风格

- **命名**：变量/函数 `camelCase` · 类型 `PascalCase` · 常量 `UPPER_SNAKE` · 文件 `kebab-case`
- **函数式优先**：纯函数 + 不可变数据，避免 class，用 `Result` 替代 try/catch
- **类型安全**：禁止 `any`，优先 `unknown` + 类型守卫，API 边界做校验

---

## 三、工程规范

- **模块职责**：单一职责、显式依赖、接口隔离
- **错误处理**：用 `Result<T, E>` 而非 throw，边界层做格式转换
- **测试**：核心逻辑必有单元测试，测试即使用示例
- **注释**：解释"为什么"而非"是什么"，TODO 带责任人

---

## 四、Git 规范

- 分支：`feat/<描述>` / `fix/<描述>` / `refactor/<描述>`
- commit：中文简洁描述做了什么
- 一个 PR 只做一件事

---

## 五、TypeScript / Bun

- 优先 `bun` 而非 npm/yarn/pnpm
- 使用 `Bun.file` / `Bun.write` 做文件 I/O
- 使用 `neverthrow` 的 `Result` 处理可能失败的逻辑
- 使用 `consola` 做统一日志输出，而非 `console.log`
