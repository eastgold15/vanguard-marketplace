# 函数式抽象模式

> 当判定需要抽象后，以下是三种逐步递进的函数式实现模式。
> 没有 OOP、没有 class、纯函数。

---

## 写法 1：简单条件分支（够用即可）

最直接的方式，逻辑简单时优先使用：

```ts
const infer = hasLocalModel()
  ? async (p: string) => localModel.run(p)
  : async (p: string) => (await fetch("/api/infer")).text()

infer("你好")
```

**适用**：只有两个实现、初始化时确定、不会动态切换。

---

## 写法 2：高阶函数封装（抽象工厂）

用高阶函数包装，解耦配置与实现：

```ts
// 具体实现
const localInfer = async (p: string) => localModel.run(p)
const remoteInfer = async (p: string) => (await fetch("/api/infer")).text()

// 高阶函数：生成统一推理函数（抽象工厂）
const createInfer = (useLocal: boolean) => {
  return useLocal ? localInfer : remoteInfer
}

// 初始化：根据环境选择实现
const infer = createInfer(hasLocalModel())

// 上层调用不变
infer("你好")
```

**适用**：实现较多、初始化逻辑复杂、纯函数式风格。

---

## 写法 3：配置映射 + 路由（多实现扩展友好）

如果后续可能要加第三、第四种实现，用对象做函数映射：

```ts
// 所有实现注册到映射表
const implMap = {
  local:  async (p: string) => localModel.run(p),
  remote: async (p: string) => (await fetch("/api/infer")).text(),
}

// 统一抽象入口
const infer = async (prompt: string) => {
  const key = hasLocalModel() ? "local" : "remote"
  return implMap[key](prompt)
}

// 上层调用
infer("你好")
```

**适用**：多实现、可动态注册，函数式/脚本风格代码（Node、前端、JS/TS、Rust、Go）。

---

## 函数式如何保证"契约"（对应 OOP 接口）

OOP 靠 `interface` 强制规范；\
函数式靠**类型签名 + 类型检查**实现同等约束：

```ts
// 定义统一函数类型（函数契约 = 函数式的接口）
type InferFn = (prompt: string) => Promise<string>

// 所有实现必须遵守这个类型，编译器强制校验
const localInfer: InferFn  = async (p) => localModel.run(p)
const remoteInfer: InferFn = async (p) => (await fetch("/api/infer")).text()

// 统一入口也复用同一份类型
const infer: InferFn = hasLocalModel() ? localInfer : remoteInfer
```

纯函数、无类、无继承，类型系统保证所有实现入参、出参、异步规则完全一致，效果 = OOP 接口。

---

## 三种写法对比

| 写法 | 复杂度 | 扩展性 | 适用场景 |
|------|--------|--------|----------|
| 条件分支 | 低 | 低 | 两个实现，初始化确定 |
| 高阶函数 | 中 | 中 | 多个实现，需要封装初始化逻辑 |
| 映射路由 | 中-高 | 高 | N 个实现，可动态注册 |
